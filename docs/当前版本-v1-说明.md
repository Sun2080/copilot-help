# 当前版本 v1 说明

这份文档用于整理仓库当前状态，给后续维护一个明确边界。

## 当前结论

当前仓库可以视为 `v1`。

这一版已经具备一个可用的中文 GitHub Copilot 实战模板库的核心骨架，不需要再为了“更完整”继续无限扩充。

更合理的方向是：

1. 先冻结当前版本
2. 在真实项目中验证
3. 基于反馈再迭代

## 现在怎么使用这套仓库

如果你现在拿到这个仓库，不建议从头到尾把所有文件都看一遍。更高效的方式是按下面顺序使用。

### 第一步：先把它当教程库看

先看这几个文件：

1. [README.md](../README.md)
2. [docs/VSCode-GitHub-Copilot-大型项目使用教程.md](VSCode-GitHub-Copilot-大型项目使用教程.md)
3. [docs/guides/如何在-VS-Code-中使用这些-Copilot-模板.md](guides/如何在-VS-Code-中使用这些-Copilot-模板.md)

这一步的目的不是记住所有文件，而是先理解这套仓库在解决什么问题：

- 怎么在 VS Code 里更系统地使用 GitHub Copilot
- Prompt、Agent、Skill、Instructions 分别适合什么场景
- 大型项目里应该怎样把 Copilot 用进工作流，而不是只拿来补全几行代码

### 第二步：再把它当模板库试用

第一次使用时，不要全量启用，先用最小组合。

建议先固定使用这 5 个：

1. `feature-plan`
2. `code-review`
3. `bugfix`
4. `pr-description`
5. `planner`

为什么先用这 5 个：

- 它们覆盖了最常见的开发动作
- 不需要你先理解所有 Agents 和 Skills
- 最容易在真实项目里形成稳定习惯

### 第三步：在 VS Code 里实际触发

你可以按下面方式理解和使用：

- Prompts：在 Chat 中输入 `/` 后选择，例如 `/feature-plan`
- Agents：在 Chat 的 Agent 选择器里切换，例如 `planner`、`reviewer`
- Skills：适合在完整工作流里使用，例如 `feature-delivery`
- Instructions：不用手动触发，它们是仓库级长期规则

简单说：

- 想做一次性任务，用 Prompt
- 想用固定角色，用 Agent
- 想跑完整工作流，用 Skill
- 想长期约束输出，用 Instructions

### 第四步：接入真实项目时怎么开始

如果你准备把这套仓库接入自己的业务项目，不要一次把所有文件都复制过去。

建议顺序是：

1. 先带上 `copilot-instructions.md`
2. 再带上 3 到 5 个高频 Prompt
3. 再根据需要引入 Agent
4. 最后再引入 Skill

最小接入组合建议是：

1. `copilot-instructions.md`
2. `feature-plan.prompt.md`
3. `code-review.prompt.md`
4. `bugfix.prompt.md`
5. `pr-description.prompt.md`

### 第五步：怎么判断它有没有价值

你可以用下面几个标准判断这套模板库对你是否真的有帮助：

1. 你是否减少了重复组织提示词的时间
2. Copilot 输出是否更稳定
3. 团队成员是否开始复用相同模板
4. 规划、审查、修复、发布这些环节是否更有标准化感觉

如果这几个问题的答案大多是“是”，那这套仓库就已经产生实际价值了。

## v1 的边界是什么

这个版本的目标是“可用”，不是“覆盖一切场景”。

所以这一版的边界应该明确为：

- 先覆盖 VS Code GitHub Copilot 的高频开发工作流
- 先覆盖中大型项目中的规划、审查、修复、测试、发布和文档场景
- 先形成一套能被真实项目接入的基础模板库

不追求：

- 一次性覆盖所有行业、所有技术栈、所有团队流程
- 在没有使用反馈的前提下继续无限新增模板

## v1 之后最合理的迭代方式

不要再按“还能补什么文件”来迭代，而要按“真实项目遇到了什么问题”来迭代。

建议按下面顺序进行：

### 第一步：试接入一个真实项目

优先只带这些内容：

1. `copilot-instructions.md`
2. `feature-plan.prompt.md`
3. `code-review.prompt.md`
4. `bugfix.prompt.md`
5. `pr-description.prompt.md`

### 第二步：试运行 1 到 2 周

观察这几件事：

- 哪些模板真的高频使用
- 哪些模板命中率低
- 哪些描述不够清楚
- 哪些规则需要更贴近真实项目

### 第三步：只基于反馈加内容

以后每新增一个模板，最好都能回答这几个问题：

1. 解决了哪个真实问题
2. 给谁用
3. 和现有模板相比新增了什么价值
4. 会不会增加使用复杂度

## 当前推荐的最小使用组合
