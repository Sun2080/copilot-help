# 当前版本 v1 说明

这份文档用于整理仓库当前状态，给后续维护一个明确边界。

## 当前结论

当前仓库可以视为 `v1`。

这一版已经具备一个可用的中文 GitHub Copilot 实战模板库的核心骨架，不需要再为了“更完整”继续无限扩充。

更合理的方向是：

1. 先冻结当前版本
2. 在真实项目中验证
3. 基于反馈再迭代

## v1 已包含什么

### 1. 教程层

当前已经具备：

- 一份完整的 VS Code GitHub Copilot 大型项目使用教程
- 一份 VS Code 中如何使用这些模板的说明
- 一份如何把这套模板接入真实项目的落地指南

### 2. 模板层

当前已经具备：

- 需求评审
- 功能规划
- 大任务拆解
- 代码审查
- Bug 修复
- 测试生成
- 接口设计
- 架构评审
- 迁移计划
- PR 描述
- 发布说明
- 回滚方案
- 日报 / 站会更新
- 复盘总结

### 3. 角色层

当前已经具备：

- `planner`
- `reviewer`
- `doc-writer`
- `release-checker`

### 4. 工作流层

当前已经具备：

- `feature-delivery`
- `release-workflow`

## v1 的边界是什么

这个版本的目标是“可用”，不是“覆盖一切场景”。

所以这一版的边界应该明确为：

- 先覆盖 VS Code GitHub Copilot 的高频开发工作流
- 先覆盖中大型项目中的规划、审查、修复、测试、发布和文档场景
- 先形成一套能被真实项目接入的基础模板库

不追求：

- 一次性覆盖所有行业、所有技术栈、所有团队流程
- 在没有使用反馈的前提下继续无限新增模板

## v1 之后最合理的迭代方式

不要再按“还能补什么文件”来迭代，而要按“真实项目遇到了什么问题”来迭代。

建议按下面顺序进行：

### 第一步：试接入一个真实项目

优先只带这些内容：

1. `copilot-instructions.md`
2. `feature-plan.prompt.md`
3. `code-review.prompt.md`
4. `bugfix.prompt.md`
5. `pr-description.prompt.md`

### 第二步：试运行 1 到 2 周

观察这几件事：

- 哪些模板真的高频使用
- 哪些模板命中率低
- 哪些描述不够清楚
- 哪些规则需要更贴近真实项目

### 第三步：只基于反馈加内容

以后每新增一个模板，最好都能回答这几个问题：

1. 解决了哪个真实问题
2. 给谁用
3. 和现有模板相比新增了什么价值
4. 会不会增加使用复杂度

## 当前推荐的最小使用组合

如果你现在就要开始用，建议先固定这组最小组合：

1. `feature-plan`
2. `code-review`
3. `bugfix`
4. `pr-description`
5. `planner`

这组已经足够覆盖大多数日常开发协作场景。

## 当前版本状态建议

建议把当前仓库状态认定为：

- `v1` 已完成
- 可以进入试用和验证阶段
- 暂时不做无目标扩展

如果后续继续迭代，建议进入 `v1.1`、`v1.2` 这种小步版本，而不是继续无限堆内容。
