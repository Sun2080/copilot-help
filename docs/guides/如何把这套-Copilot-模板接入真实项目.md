# 如何把这套 Copilot 模板接入真实项目

这篇文档不再解释“模板是什么”，而是直接回答一个更实际的问题：如果你已经有自己的业务项目，怎么把这个仓库里的 Copilot 模板真正接进去。

## 目标

你要做的不是把所有文件一次性复制过去，而是按优先级分阶段引入，让团队能用、愿意用、结果也稳定。

## 推荐接入顺序

### 第 1 步：先接项目级规则

先把下面这类内容接入你的业务仓库：

- `copilot-instructions.md`
- 与文档或定制文件相关的 `instructions`

原因：

- 先把长期规则固定下来
- 减少每个人和 Copilot 交互时的随机性

### 第 2 步：只挑 3 到 5 个高频 prompts

不要一次全部搬过去。优先挑这些最常用的：

1. `feature-plan`
2. `code-review`
3. `bugfix`
4. `pr-description`
5. `task-breakdown`

原因：

- 这些最容易形成稳定习惯
- 也是大型项目里最常重复的工作

### 第 3 步：再引入 agents

等团队已经习惯用 prompts 之后，再引入：

- `planner`
- `reviewer`
- `doc-writer`
- `release-checker`

原因：

- Agent 更像固定角色
- 需要团队对使用场景有更清晰认知

### 第 4 步：最后再引入 skills

Skill 更像完整工作流，适合在下面场景使用：

- 中大型功能交付
- 高风险发布准备

建议最后再引入：

- `feature-delivery`
- `release-workflow`

## 不同项目的接入建议

### 单仓库项目

最简单，直接把 `.github/` 目录放到仓库根目录即可。

### Monorepo

建议：

- 先在根目录放全局规则
- 再针对不同包或服务增加更具体的 instructions
- 必要时打开父仓库自定义发现能力

### 团队多人协作项目

建议：

- 先由一个人整理规则
- 再在真实任务里试跑 1 到 2 周
- 最后再统一调整文件命名、描述和模板内容

## 如何判断接入是否有效

至少看这 4 件事：

1. 团队是否真的开始复用 prompt
2. Copilot 输出是否更稳定
3. 是否减少了重复解释规则的次数
4. 是否能在代码审查和发布阶段形成标准化输入输出

## 最常见的错误接法

### 错误 1：一次性全搬过去

问题：

- 团队不知道什么时候该用哪个模板
- 实际使用率会很低

### 错误 2：只有模板，没有项目规则

问题：

- 每个人提示词风格差异很大
- Copilot 输出波动明显

### 错误 3：只有规则，没有工作流

问题：

- 团队知道要遵守什么，但不知道怎么触发实际工作流程

## 推荐的最小落地方案

如果你现在就要在一个真实项目里试运行，建议最小落地组合是：

1. `copilot-instructions.md`
2. `feature-plan.prompt.md`
3. `code-review.prompt.md`
4. `bugfix.prompt.md`
5. `pr-description.prompt.md`

这套组合足够覆盖大多数日常开发环节。

## 最后建议

接入 Copilot 模板库，不要追求“文件数量多”，而要追求“团队在真实工作里真的会用”。

先让 3 个模板变成日常习惯，再慢慢扩到 10 个模板，效果会更稳。
