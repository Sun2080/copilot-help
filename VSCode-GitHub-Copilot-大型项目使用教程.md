# VS Code GitHub Copilot 大型项目使用教程

## 先看这页

如果你时间不多，先记住 8 条结论：

1. 大型项目不要一上来就让 Copilot 直接写代码，先用 Ask 或 Plan 建立上下文。
2. 真正高效的模式不是“让它多写”，而是“让它在清晰边界内持续交付并自证”。
3. 复杂需求一律拆成小任务，每个子任务都要能单独验证。
4. Agent 适合执行，Ask 适合理解和评审，Plan 适合拆解和控风险。
5. 每次提示词至少写清楚：目标、范围、约束、参考对象、验证方式、输出格式。
6. 不要只让它生成代码，要要求它跑测试、跑构建、总结风险。
7. 项目级 Instructions、Prompt Files、Custom Agents、MCP 才是大型项目真正的提效杠杆。
8. 你最终要追求的不是“AI 自动开发”，而是“AI 参与工程化开发流程”。

---

## 这份教程适合谁

这份教程适合这几类人：

- 已经在 VS Code 里装了 GitHub Copilot，但大多只会看补全，不会系统用 Chat、Agent、Edits、Prompt、Instructions。
- 想把 Copilot 从“代码补全工具”升级成“项目开发助手”。
- 正在做中大型项目，希望让 Copilot 参与需求拆解、脚手架搭建、跨文件修改、调试、测试、重构、文档整理和 PR 准备。
- 希望在团队内把 Copilot 用成标准化工作流，而不是每个人各聊各的。

这份教程的目标不是教你“怎么点按钮”，而是教你建立一套高效率、可复制、可审查的工程工作流。

## 一、先建立正确认知

很多人把 GitHub Copilot 理解成“自动补全”。这是老用法。

现在 VS Code 里的 GitHub Copilot，核心能力已经变成了 3 层：

1. 行内补全
2. 对话式修改
3. Agent 式端到端开发

你可以把它理解为三档能力：

- 最轻量：你写代码时，它补几行。
- 中量级：你告诉它改哪里，它在当前文件或局部范围内改。
- 最重度：你给它一个目标，它自己规划、跨文件修改、执行命令、跑测试、根据结果继续修。

如果你要高效开发大型项目，真正要学会的是第 2 档和第 3 档。

更准确地说，Copilot 现在像一个“有上下文、有工具、有执行能力的协作工程师”，但它仍然不是项目负责人。真正的职责划分应该是：

- 你负责定义目标、边界、验收标准和架构约束。
- Copilot 负责分析、生成、联动修改、验证和整理结果。
- 你和 Copilot 共同完成迭代闭环。

这套分工一旦建立，效率会明显高于把它当作“更聪明的搜索框”。

## 二、VS Code 里 Copilot 的核心能力

### 1. Inline Suggestions

就是你打字时自动出现的补全建议。

适合场景：

- 写函数骨架
- 写重复逻辑
- 生成单元测试样板
- 写 DTO、类型定义、接口映射

它适合“你已经知道要写什么，只想提速”。

不适合：

- 不清楚需求
- 需要跨文件联动
- 需要理解架构再动手

### 2. Inline Chat

快捷键通常是 Ctrl+I。

它直接在编辑器当前位置工作，适合：

- 局部重构
- 解释某段代码
- 把一段代码改成另一种风格
- 给当前函数补测试
- 修一处明确的小 bug

一句话：

Inline Chat 适合“就地修改当前代码”。

### 3. Chat View

快捷键通常是 Ctrl+Alt+I。

这是主战场。它适合：

- 多轮对话
- 多文件修改
- 项目分析
- 需求拆解
- 调试流程
- 让 Agent 自动干活

你做大型项目，主要时间会花在这里。

### 4. Quick Chat

快捷键通常是 Ctrl+Shift+Alt+L。

适合临时问答，不打断当前编辑节奏。

例如：

- 这个错误是什么意思？
- 这个类是做什么的？
- 给我一个正则表达式
- 帮我写个 curl 命令

### 5. Agent 模式

这是近阶段最值得学习的能力。

Agent 不只是回答问题，而是可以：

- 分析代码库
- 制定步骤
- 修改多个文件
- 执行终端命令
- 跑构建和测试
- 根据报错继续修复

如果你在做大型项目，Agent 才是生产力放大器。

### 6. Plan / Ask / Agent 三种模式

VS Code 当前内置的典型工作模式可以理解为：

- Ask：只问答，不改代码
- Plan：先分析并产出实施计划
- Agent：直接执行任务

建议你这样用：

- 需求不清楚时，用 Ask
- 任务大、牵涉多模块时，先用 Plan
- 方案确定后，再切到 Agent 执行

这比一上来就让 AI 乱改代码稳定得多。

### 7. Copilot Edits

它是“让 Copilot 批量编辑多个文件，但你逐步审阅”的关键能力。

适合：

- 一次性改接口命名
- 批量补日志
- 同步更新 service、controller、test、docs
- 迁移框架 API

它的价值在于：

- 改得快
- 你还能保留审查权
- 比纯聊天描述更可控

### 8. Smart Actions

VS Code 里已经有不少现成的 AI 动作，例如：

- 生成 commit message
- 修测试失败
- 错误修复
- 语义搜索
- 解释代码

这类功能很适合“日常提效”，虽然不如 Agent 显眼，但长期收益很高。

### 9. Next Edit Suggestions

这是最近很值得用的一类能力。

它不只是补下一行代码，而是预测“你接下来还会改哪里”。

适合：

- 修改一个接口后，继续补另一个关联位置
- 改完类型后，自动建议你同步改使用方
- 重构过程中连续推进多个小变更

它会明显减少来回跳文件的次数。

## 三、先学会选模式，而不是盲目发提示词

大多数低效使用 Copilot 的根因，不是提示词写得太短，而是模式选错了。

### 模式选择表

| 你的目标 | 最适合的模式 | 原因 |
| --- | --- | --- |
| 看懂陌生模块 | Ask | 先建立认知，不要提前修改 |
| 拆一个大需求 | Plan | 先做任务边界和风险识别 |
| 执行一个子任务 | Agent | 能跨文件改动并联动验证 |
| 局部重构一个函数 | Inline Chat | 修改范围小，反馈快 |
| 批量同步修改多个文件 | Copilot Edits | 审查友好、适合联动改动 |
| 临时问一个小问题 | Quick Chat | 不打断当前工作流 |
| 连续写模板化代码 | Inline Suggestions | 最轻量、最顺手 |

### 一个简单判断法

如果你还说不清楚“应该改哪些文件”，先别进 Agent，先 Ask 或 Plan。

如果你已经明确知道：

- 改动边界是什么
- 参考代码在哪里
- 验收标准是什么

这时再进 Agent，成功率会高很多。

## 四、最近值得你重点关注的新能力

下面这些能力，基本决定了 Copilot 现在是否能真正参与大型项目开发。

### 1. Agent Sessions

你现在可以把任务作为一个持续会话来运行，而不是一问一答后就结束。

优势：

- 会话有上下文
- 可以暂停、恢复、切换
- 可以并行跑多个任务
- 适合大型项目里的多子任务协同

例如你可以同时开几个会话：

- 会话 A：实现用户登录
- 会话 B：修复测试失败
- 会话 C：补接口文档

### 2. Session Type

现在 Agent 不只在本地执行，通常还可以选择不同运行方式：

- Local：本地交互式执行，适合探索和即时反馈
- Background：在本机后台自动跑，适合耗时任务
- Cloud：云端执行并产出 PR，适合团队协作
- Third-party：使用第三方模型或外部 Agent

建议：

- 架构探索、快速试错，用 Local
- 长时间重构、批量生成、跑测试，用 Background
- 明确边界的任务，交给团队流程，用 Cloud

### 3. Autopilot（预览）

这是高自动化模式。

它会：

- 自动批准工具调用
- 自动回答一些运行时问题
- 持续执行直到任务完成

优点：

- 省掉频繁确认
- 很适合明确、边界清晰的任务

风险：

- 误改范围可能扩大
- 如果提示词不清楚，可能走偏

所以建议：

- 小任务、探索期，不要先开 Autopilot
- 任务边界明确、有测试兜底时，再考虑开启

### 4. Custom Instructions

这是项目级效率的基础设施。

你可以让 Copilot 长期记住：

- 项目架构规则
- 命名规范
- 测试要求
- 提交标准
- 日志和异常处理规范
- 前后端接口约束

在大型项目里，不写 Instructions，Copilot 产出会越来越飘。

### 5. Prompt Files

适合把重复性工作沉淀成模板。

例如：

- 新增 REST 模块模板
- 新增 React 页面模板
- 为某个服务补测试模板
- 生成 PR 描述模板

它让“常见工作流”从临时聊天，升级成可复用资产。

### 6. Custom Agents / Skills / MCP / Hooks

这是从“个人提效工具”走向“团队自动化平台”的关键。

它们的作用大致可以这么理解：

- Custom Agents：给 AI 定角色，例如架构师、代码审查员、安全检查员
- Skills：把复杂能力打包成可复用技能
- MCP Servers：接外部系统，例如数据库、API、文档系统、工单系统
- Hooks：在关键节点执行自动命令，例如格式化、lint、策略检查

如果你的项目规模大、团队多人协作，这部分非常值得投入。

### 7. Chat Debug View / Agent Logs

如果你发现 Copilot：

- 理解错上下文
- 没发现某些文件
- 指令没生效
- 工具调用异常

不要靠猜。

直接看 Agent Logs 和 Chat Debug View，能看到：

- 它收到的上下文
- 它发现了哪些指令文件
- 它执行了哪些工具
- 为什么生成这个结果

这是排查“为什么 AI 不听话”的核心能力。

## 五、先把环境配置到位

### 1. 必备前提

你至少需要：

- 已登录 GitHub 账号
- 已开通 GitHub Copilot 权限
- VS Code 更新到较新版本
- Git、Node、Python、你项目需要的运行环境都能在本机正常工作

因为 Agent 真正强大，不只是生成代码，而是会调用工具、跑命令、检查结果。

### 2. 建议打开的 VS Code 视图

建议把这些视图固定到工作流里：

- Chat
- Source Control
- Problems
- Terminal
- Testing
- Explorer

原因很简单：

Copilot 的价值不是“回答”，而是“回答后立刻改、改后立刻验证”。

### 3. 第一次进入项目后先做这件事

在 Chat 里执行：

```text
/init
```

这通常会帮助你初始化项目的 Copilot 指令文件。

然后你再补充项目规范，例如：

- 后端必须先写接口定义再写实现
- 所有新接口必须带单元测试和错误处理
- React 组件使用函数式组件
- 数据访问统一走 repository 层
- 日志统一使用某个 logger

这一步会极大提升后续回答质量。

## 六、30 分钟上手路线

如果你今天就想把 Copilot 用起来，不要先研究所有高级功能，按这个顺序做：

### 第 1 步：让它先解释项目

```text
先分析当前项目结构，不要修改代码。
告诉我：目录分层、核心模块、启动入口、测试入口，以及新增功能应该落在哪些目录。
```

### 第 2 步：执行 /init

让项目有最基本的 Instructions，避免每次都重复介绍风格和约束。

### 第 3 步：挑一个中小任务

不要挑“做完整个支付系统”，而要挑“补一个接口、补一组测试、修一个构建错误”这种 30 到 90 分钟可完成的任务。

### 第 4 步：先 Plan 再 Agent

```text
先给出实施计划，不要改代码。
确认计划后，再只执行第 1 步。
```

### 第 5 步：要求它自证

```text
改完后运行相关测试或构建，并汇报结果、修改文件和剩余风险。
```

只要你先把这 5 步养成习惯，使用效果就会和“随便聊聊”拉开很大差距。

## 七、做大型项目时，推荐的最佳工作流

下面这套流程，比“想到什么问什么”效率高很多。

### 阶段 1：先让 Copilot 理解项目

不要一上来就说“帮我写功能”。

先让它建立项目心智模型。

建议先问：

```text
先阅读当前项目结构，告诉我：
1. 目录分层和职责边界
2. 核心业务模块
3. 关键技术栈
4. 配置、构建、测试入口
5. 你认为最适合新增功能的位置
不要修改代码，先输出分析结论。
```

这里建议用 Ask 或 Plan。

目的：

- 先让它理解架构
- 先发现约束
- 先减少乱改

### 阶段 2：让它先出计划，不要直接写

对大型需求，推荐先用 Plan：

```text
我要新增“订单退款审批流”功能。
请先不要写代码。
先基于现有项目结构输出实施计划，包括：
1. 需要改动的模块
2. 新增/修改的数据结构
3. API 设计
4. 前端页面与状态流
5. 测试范围
6. 风险点与兼容性问题
```

为什么先计划：

- 你能先纠偏
- 避免它直接生成错误结构
- 便于拆任务并行执行

### 阶段 3：把大任务拆成可验证的小任务

错误用法：

```text
帮我做完整个支付系统
```

正确用法：

```text
根据刚才计划，先只实现后端退款申请接口。
要求：
1. 复用现有订单模块风格
2. 新增 DTO、service、controller、test
3. 保持现有错误码规范
4. 写完后运行相关测试
5. 输出你修改了哪些文件和验证结果
```

这是大型项目里最重要的一条原则：

把 AI 当作可控执行单元，不要当作会读心的全自动工程师。

### 阶段 4：明确上下文范围

现在的 Copilot 支持显式上下文，你应该主动用。

常见方式包括：

- 指定文件
- 指定符号
- 指定目录
- 指定整个代码库
- 指定终端输出

典型提示：

```text
参考 #codebase、当前打开的退款模块、订单 service 和相关测试。
只在退款相关目录内工作，不要修改共享基础设施代码。
```

上下文越准，结果越稳。

### 阶段 5：让它改，且要求它自证

不要只说“实现它”。

要把验证动作也一起交代：

```text
实现这个功能后：
1. 运行相关测试
2. 如果失败，继续修复直到通过
3. 总结改动点
4. 列出风险和未覆盖项
```

这会把 Copilot 从“代码生成器”变成“带验证的执行者”。

### 阶段 6：你负责审查关键点，不要盲收

你要重点检查：

- 接口边界
- 类型定义
- 错误处理
- 事务和并发
- 权限校验
- 测试是否真覆盖关键路径
- 是否改到了不该改的地方

记住：

AI 可以完成大量工作，但架构边界、业务正确性、隐性约束，最终还是需要你把关。

## 八、一套适合大型项目的 Agent 执行 SOP

如果你想把 Copilot 用出“自动化开发”的感觉，建议固定采用下面这套 SOP。

### Step 1：定义任务边界

至少明确这 5 件事：

- 改什么
- 不改什么
- 参考什么
- 怎么验收
- 失败后怎么办

### Step 2：确认执行级别

可以按风险分三级：

- L1 低风险：局部代码、补测试、改文档、修 lint
- L2 中风险：跨文件功能、批量重构、接口联动
- L3 高风险：数据库 schema、权限系统、基础设施、核心交易链路

建议：

- L1 可以直接 Agent
- L2 先 Plan 再 Agent
- L3 先 Plan，再拆成多个 L1/L2 子任务，必要时人工主导

### Step 3：先执行最小闭环

不要一口气把所有子任务都交出去。先选一个最小闭环，比如：

- 一个后端接口
- 一组测试
- 一个页面模块
- 一次枚举迁移

先验证这一块做对了，再扩大范围。

### Step 4：让 Agent 输出可审查结果

要求它每次都汇报：

- 修改文件
- 修改原因
- 运行过哪些命令
- 测试或构建结果
- 剩余风险

### Step 5：人工验收后再继续下一步

这个节奏很像小步快跑，而不是一键全自动。大型项目最怕的不是慢，而是大范围错误改动后返工。

## 九、大型项目里最有效的 8 种 Copilot 用法

### 1. 新功能脚手架

用法：

先让它分析现有模块模式，再生成一个同风格新模块。

提示模板：

```text
阅读用户模块的实现风格，然后为“通知中心”生成同结构模块。
要求复用相同的目录分层、命名风格、错误处理和测试组织方式。
先列出计划，再实施。
```

### 2. 跨文件重构

用法：

重命名、接口迁移、拆分大文件、替换旧 API。

提示模板：

```text
将旧的 PaymentStatus 枚举迁移到新的领域模型定义。
请更新所有引用点、测试和序列化逻辑。
修改后运行构建并报告受影响范围。
```

### 3. 接手陌生代码

用法：

让它快速建立认知。

提示模板：

```text
分析这个仓库的认证与授权链路。
按“入口 -> 中间层 -> 数据层 -> 测试”说明。
最后告诉我如果要加 RBAC，最合理的扩展点在哪里。
```

### 4. 调试失败测试

用法：

把错误栈、测试输出、相关代码一起给它。

提示模板：

```text
根据当前测试失败信息定位根因。
只修复根因，不要顺手重构无关代码。
修复后重新运行相关测试，并解释为什么之前会失败。
```

### 5. 生成测试

用法：

让它补回归测试、边界测试、异常测试。

提示模板：

```text
为这个 service 补完整测试，至少覆盖：
1. 正常路径
2. 参数非法
3. 外部依赖失败
4. 并发/重复提交保护
不要修改生产代码，除非测试暴露出真实缺陷。
```

### 6. 文档和交接

用法：

让它整理：

- 模块说明
- API 文档
- 发布说明
- 迁移说明
- PR 描述

提示模板：

```text
基于本次改动，生成一份给团队的变更说明。
包含：背景、改动模块、兼容性影响、测试结果、回滚建议。
```

### 7. 代码审查前自检

用法：

在提交 PR 前让它做一轮检查。

提示模板：

```text
请从代码审查视角检查这次改动。
重点看：
1. 潜在 bug
2. 回归风险
3. 漏掉的测试
4. 命名和职责边界
5. 性能与安全问题
不要重写代码，先列出问题清单。
```

### 8. 自动化重复流程

用法：

把经常重复的操作沉淀成 Prompt Files、Instructions、Skills、Hooks。

这一步是“个人使用 Copilot”和“团队级使用 Copilot”的分水岭。

## 十、你必须学会的提示词写法

提示词质量，直接决定结果质量。

一个高质量提示词通常要包含 6 个元素：

1. 目标
2. 范围
3. 约束
4. 参考对象
5. 验证方式
6. 输出格式

### 差的写法

```text
帮我改一下登录功能
```

问题：

- 目标不清楚
- 范围不清楚
- 没有约束
- 没有验证要求

### 好的写法

```text
在现有认证模块中新增“登录失败次数限制”能力。
要求：
1. 仅修改认证相关代码
2. 复用现有缓存和日志设施
3. 连续失败 5 次后锁定 15 分钟
4. 保持现有 API 响应结构不变
5. 补充单元测试和集成测试
6. 修改后运行相关测试并汇报结果
```

### 更好的写法

```text
请先分析认证模块当前结构，再实现“登录失败次数限制”。
参考现有用户登录接口、缓存封装、错误码定义和测试风格。
限制条件：
1. 不修改公共基础库
2. 不改变数据库 schema，优先用缓存实现
3. 保持现有 controller 响应契约不变
4. 必须补测试
5. 运行相关测试
输出：
1. 先给出计划
2. 再实施
3. 最后总结修改文件、测试结果、剩余风险
```

### 一个可复用的提示词公式

你可以长期复用下面这个公式：

```text
目标：我要完成什么。
范围：允许改哪些目录或文件。
禁区：不要改哪些模块，不要做哪些动作。
参考：优先参考哪些现有实现。
验证：必须运行什么测试、构建或检查。
输出：最后按什么格式汇报。
```

例如：

```text
目标：新增退款申请接口。
范围：仅修改 refund 和 order 相关目录。
禁区：不要改共享基础设施，不要新增依赖，不要改 schema。
参考：参考现有 cancelOrder 接口与测试风格。
验证：运行 refund 相关测试与后端构建。
输出：按“修改文件 / 验证结果 / 风险”格式汇报。
```

这就是大型项目场景里的标准写法。

## 十一、如何让 Copilot 在大型项目里更稳

### 1. 永远先给边界

告诉它：

- 改哪些目录
- 不改哪些目录
- 哪些公共模块不能碰
- 是否允许改 schema
- 是否允许装依赖

### 2. 永远要求先分析再执行

尤其是：

- 新人接手旧项目
- monorepo
- 微服务
- 多端联动
- 老系统重构

### 3. 永远要求验证

至少要求它：

- 跑测试
- 跑构建
- 解释失败原因
- 说明未覆盖风险

### 4. 大任务必须拆分

一条经验：

如果一个任务涉及 5 个以上文件、2 个以上模块、还牵涉测试和文档，那就应该先拆。

### 5. 把项目规范写进 Instructions

不要每次聊天都重新描述：

- 命名规范
- 分层原则
- 异常处理
- 日志规范
- 测试最低要求
- 安全规则

这些应该固化。

### 6. 用 Plan 模式做需求拆解

Plan 模式非常适合：

- 做任务方案
- 输出里程碑
- 找出风险点
- 估算影响范围

### 7. 用 Agent 模式做执行

Agent 模式适合：

- 跨文件修改
- 跑命令
- 验证结果
- 自动修复

### 8. 用 Ask 模式做认知和审查

Ask 模式适合：

- 理解项目
- 解释代码
- 评审方案
- 让它指出风险

## 十二、一套适合大型项目的日常工作流

下面是一套我建议你长期采用的固定节奏。

### 上午开始开发前

1. 打开 Chat，回顾昨天未完成任务
2. 让 Copilot 总结当前分支改动目的
3. 让它扫描待处理问题、失败测试、TODO
4. 明确今天只做哪一个模块

提示词示例：

```text
基于当前分支和工作区状态，总结我今天应该优先处理的任务。
请按“高优先级 bug / 功能开发 / 技术债 / 风险检查”分类输出。
```

### 开始做一个新功能时

1. 先 Ask 或 Plan 分析
2. 再拆任务
3. 再 Agent 执行一小块
4. 每做完一块就审查 diff 和测试结果

### 开始做重构时

1. 先让它列影响范围
2. 再选 Copilot Edits 或 Agent 模式
3. 先改一批核心文件
4. 先验证，再扩大范围

### 提交前

1. 让它生成 commit message
2. 让它做一次代码审查
3. 让它生成 PR 描述
4. 让它总结风险和测试覆盖情况

## 十三、团队里怎么把 Copilot 用到“自动化开发”层级

如果你是个人开发者，用到 Chat 和 Agent 已经很强。

但如果你是团队或大型项目，建议继续往下做。

### 1. 建立项目级 Instructions

把这些写进去：

- 技术栈规范
- 代码风格
- 目录职责
- 测试要求
- API 设计规范
- 安全约束
- 禁止修改区域

### 2. 建立 Prompt Files

把常见任务模板化，例如：

- 新增接口
- 新增页面
- 修 bug
- 写测试
- 做 code review
- 生成 PR 说明

### 3. 建立自定义 Agent

典型角色：

- 架构审查 Agent
- 安全审查 Agent
- 测试生成 Agent
- 文档整理 Agent
- 发布检查 Agent

### 4. 接入 MCP

当你的项目已经不只是一堆代码时，MCP 就有价值了。

它可以让 Copilot 连接：

- 数据库
- 内部 API
- Swagger/OpenAPI
- 知识库
- 工单系统
- CI/CD 信息

这样 Copilot 不只是“看代码”，而是“能利用整个工程环境”。

### 5. 用 Hooks 做流程自动化

例如：

- 编辑后自动跑 formatter
- 提交前自动跑 lint
- 关键目录修改时自动做安全检查
- 生成代码后自动补许可证或头部规范

这会让 AI 开发更接近工程化流水线，而不是聊天玩具。

## 十四、成熟度路线图

你可以把团队使用 Copilot 的成熟度分成 4 个阶段。

### 阶段 1：个人提效

- 主要在用补全和简单聊天

表现：

### 阶段 3：团队规范化

## 十五、你最容易踩的坑

### 1. 一上来就让它写完整系统

后果：

- 结果看起来很多
- 但结构容易错
- 返工成本很高

改法：

- 先 Plan
- 再拆成可验证子任务

### 2. 不给上下文

后果：

- 它按通用经验乱猜
- 和你项目风格不一致

改法：

- 指定目录、文件、符号、代码库上下文和参考实现

### 3. 不给约束

后果：

- 它可能装新依赖
- 改 schema
- 改公共模块
- 破坏兼容性

改法：

- 明确禁区和允许动作

### 4. 不要求验证

后果：

- 代码看起来像能跑
- 实际没测试过

改法：

- 每次都要求跑测试、构建或至少做静态检查

### 5. 盲目接受所有修改

后果：

- 引入隐性 bug
- 破坏架构边界
- 增加技术债

改法：

- 把重心放在审查 diff 和验收标准上

### 6. 不沉淀自定义规则

后果：

- 每次都要重讲
- 团队成员用法不一致
- AI 结果波动大

改法：

- 逐步沉淀 Instructions、Prompt Files、Custom Agents

## 十七、给你一套可以直接复制的提示词模板

### 1. 先分析项目

```text
先分析当前仓库，不要修改代码。
请告诉我：
1. 目录结构与职责分层
2. 核心业务模块
3. 启动、构建、测试入口
4. 现有代码风格与架构约束
5. 如果我要新增 [功能名]，最合理的落点在哪里
```

### 2. 先出计划

```text
我要实现 [功能名]。
先不要写代码。
请先给出实施计划，包含：
1. 受影响模块
2. 需要新增/修改的数据结构
3. API 或页面改动
4. 测试计划
5. 风险点
6. 推荐拆分顺序
```

### 3. 执行一个子任务

```text
根据刚才的计划，先只实现第 1 步。
要求：
1. 仅修改 [目录/模块]
2. 保持现有编码风格
3. 补充必要测试
4. 修改后运行相关测试
5. 输出修改文件、测试结果、剩余风险
```

### 4. 修 bug

```text
根据当前报错和相关代码定位根因。
要求：
1. 只修复根因，不做无关重构
2. 保持对外接口不变
3. 增加回归测试
4. 运行相关测试并汇报结果
```

### 5. 做代码审查

```text
请从代码审查角度检查当前改动。
重点关注：
1. 逻辑 bug
2. 回归风险
3. 漏掉的测试
4. 性能问题
5. 安全问题
先列问题，不要直接改代码。
```

### 6. 生成 PR 描述

```text
根据当前改动生成 PR 描述。
包括：
1. 背景
2. 改动内容
3. 测试结果
4. 风险与兼容性
5. 回滚方案
```

### 7. 让它接手一个大型任务的开场白

```text
你现在作为当前项目的协作开发者，先不要直接修改代码。
请先分析代码结构、模块边界、测试入口和相关实现风格。
然后把 [任务名] 拆成 3 到 7 个可验证步骤。
每一步说明：目标、受影响文件、风险、验证方式。
等我确认后，你再只执行第 1 步。
```

## 十八、一个真实的大型项目 Copilot 使用范式

假设你要开发“企业后台的退款审批流”。

不要这样做：

```text
帮我把退款审批流做完
```

应该这样做：

### 第一步：分析

```text
分析订单、退款、审批相关模块，说明当前结构和扩展点。
不要修改代码。
```

### 第二步：计划

```text
基于现有结构，给出退款审批流的实施计划。
覆盖数据模型、接口、页面、权限、测试和回滚考虑。
```

### 第三步：拆任务

把它拆成：

- 后端退款申请接口
- 审批状态机
- 权限校验
- 管理端审批页面
- 通知逻辑
- 单元测试与集成测试
- 文档与 PR

### 第四步：逐块执行

```text
先只实现“后端退款申请接口”。
只改退款域相关目录，复用现有异常处理和日志风格。
补测试，跑测试，最后汇报。
```

### 第五步：每一步都审查

看：

- diff 是否越界
- 测试是否真实覆盖
- 命名是否符合领域模型
- 是否引入了不必要依赖

### 第六步：提交前自检

```text
对本次退款申请接口改动做代码审查，给出问题清单和改进建议。
```

这套流程的本质是：

不是让 AI 一次性替你做完整项目，而是让 AI 持续承担“分析、执行、验证、整理”中的高重复劳动。

## 十九、一个推荐的项目目录沉淀思路

当你开始长期使用 Copilot，建议逐步把项目内与 AI 相关的资产沉淀出来。常见思路是：

- 项目级指令：放全局编码规范、测试要求、架构边界
- 文件级指令：针对某一类目录或语言补充专项规则
- Prompt Files：放高频任务模板
- Agents：放特定角色能力
- Skills：放跨场景复用工作流

你不需要第一天就把这些都建完，但要有“把高频经验沉淀成资产”的意识。

## 二十、最实用的结论

如果你只记住几句话，记住下面这些就够了。

1. 大型项目里，先 Plan，后 Agent。
2. 永远给边界、约束、参考对象和验证要求。
3. 大任务必须拆小，逐块验收。
4. 用 Instructions 固化规范，用 Prompt Files 固化流程。
5. 用 Agent 干执行，用 Ask 干分析和审查。
6. 不要只看它写得快，更要看它是否验证过。
7. 真正的高效，不是“让 AI 写更多”，而是“让 AI 在正确边界内持续交付可验证结果”。
